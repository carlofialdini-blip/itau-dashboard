## 1. Project Overview

- **Name:** Portfolio Intelligence Dashboard (internal working name; header reads "Portfolio Intelligence / Market Monitor")
- **Owner:** Carlo Fialdini (vittorio.fialdini@gmail.com), works at Itaú
- **Purpose:** Aggregate news, market data, and economic indicators for a tracked portfolio of Brazilian/global companies, plus dedicated Brazil, China, and Credit-market views. Built for internal team distribution — not a live web app.
- **Distribution model:** Generated **twice a day** into a single static `dashboard.html` file, then uploaded to SharePoint / sent directly to teammates. Recipients just open the file in a browser — no server, no install, no network calls at view time. This shaped almost every architectural decision below.
- **Tech stack:** Python 3 (stdlib + a handful of libs, see §5), Jinja2 templating, vanilla JS, Plotly.js (via CDN in the template). No frontend build step, no npm/Node runtime dependency for the app itself, no database.
- **Package manager:** pip, `requirements.txt` at repo root.
- **Repo:** `git@github.com:carlofialdini-blip/itau-dashboard.git` (GitHub account `carlofialdini-blip`, distinct from `carlofialdini` — SSH key is tied to the `-blip` account).

## 2. Architecture

### Folder structure
```
update_dashboard.py       # ENTRY POINT — run twice a day, refreshes everything
setup.py                  # ENTRY POINT — one-time per-machine bootstrap (pip install + first run)
requirements.txt
dashboard.html             # generated output (gitignored)
assets/
  itau_logo.png            # 120x120 processed logo, embedded as base64 into dashboard.html
Itaú.jpeg                  # original source logo (3840x3840), kept for future reprocessing
data/                      # generated caches (gitignored except portfolio.xlsx)
  portfolio.xlsx           # source list of tracked companies — columns: Company, Sector, Keywords, Ticker (optional, blank-safe)
  news_cache.json           china_news_cache.json
  brazil_news_cache.json    credit_news_cache.json
  gdelt_news_cache.json     # portfolio companies, GDELT-sourced (complements news_cache.json)
  events.json               china_events.json
  brazil_events.json
  br_ettj_history.json      # accumulating Brazil 10Y yield history (ANBIMA), grows one point/day
templates/
  dashboard_template.html   # single Jinja2 template for the entire static site (5 pages)
core/
  generate_dashboard.py     # main builder: reads caches, fetches live chart data, renders template
  net_utils.py               # shared get_with_retry() HTTP helper (retry + backoff)
  scoring.py                  # shared importance_bucket() (Low/Medium/High)
  trusted_sources.py           # shared TRUSTED_DOMAINS/TRUSTED_DOMAINS_EN whitelists + is_trusted()/outlet_name()
scrapers/                   # news scrapers, one per source (not per page — see below)
  scraper.py                 # portfolio companies (from portfolio.xlsx), Google News RSS
  gdelt_scraper.py            # portfolio companies, GDELT DOC 2.0 API — PT+EN, complements scraper.py
  brazil_scraper.py          china_scraper.py          credit_scraper.py     # Google News RSS
  fuel_scraper.py             # ANP open-data (liquidos.zip) → Energy sub-page's dataset, not news
events/                     # event-calendar generators, one per source
  events_generator.py        # earnings (yfinance) + COPOM (hardcoded) + IPCA/IBGE (API)
  brazil_events_generator.py # IBGE calendar + COPOM + BCB Focus Report (weekly Mondays)
  china_events_generator.py  # NBS/PBOC release dates (pattern-computed, no network) + conferences (hardcoded)
```

### Data flow (what `update_dashboard.py` runs, in order)
1. **Scrapers** (`scrapers/*.py`) — query Google News RSS (or, for `gdelt_scraper.py`, the GDELT DOC 2.0 API) per sector/company, score relevance, merge with existing cache → write `data/*_cache.json`. **Freshness policy differs by source**: `brazil_scraper.py` is strict same-day-only (Brasília calendar date), with per-keyword querying (not one OR-mega-query — see §4) and score-based backfill to a 30-article-per-sector floor; `scraper.py`/`china_scraper.py`/`credit_scraper.py` keep a 7-day day-balanced window (today/yesterday/older fallback buckets); `gdelt_scraper.py` keeps a 7-day window too but has no day-balancing (GDELT's own `sort:datedesc` already gives recency-first results, no relevance-ranking cap to correct for).
2. **Event generators** (`events/*.py`) — build forward-looking event calendars (mix of live APIs and hardcoded annual schedules) → write `data/*_events.json`.
3. **`scrapers/fuel_scraper.py`** — downloads ANP's `liquidos.zip` open-data file, parses ~1M rows of company-level fuel-distribution sales, aggregates into the Energy sub-page's `{E,P,R,M,T,G,K}` shape → writes `data/fuel_data_cache.json`. Not news or an events calendar (hence not grouped under 1/2 above), but wired into `update_dashboard.py`'s `STEPS` the same fault-isolated way — its own subprocess, own timeout, and a failure here just means the Energy page keeps its last-cached snapshot rather than blocking the rebuild.
4. **`core/generate_dashboard.py`** — reads all cache files + `portfolio.xlsx`, fetches *live* chart data (BCB OLINDA API for Brazil/Credit macro series, chinadata.live API for China datasets, yfinance + BCB + ANBIMA for the Cockpit market overview), **merges all 4 news sources and all 3 event sources into two unified feeds** (`load_unified_news()`, `load_unified_events()` — see Architecture below), renders `templates/dashboard_template.html` via Jinja2 → `dashboard.html`. Every chart's data is embedded inline as JSON in a `<script>` block — the file has zero `fetch()`/XHR calls, so it opens correctly via `file://` with no server.
5. **`update_dashboard.py`** orchestrates 1–4 via `subprocess`, **continues past individual step failures** (a network hiccup in one scraper doesn't block the rebuild — that source just keeps its last-cached data), 600s timeout per step.
6. **`setup.py`** = `pip install -r requirements.txt` then delegates to `update_dashboard.py`. Run once per machine (Carlo's Mac, the work Windows PC) — never sent to teammates.

### Pages in `dashboard.html`
As of the July 2026 reorganization, the dashboard is organized by **content type**, not by source — one News page instead of four separate news feeds, one Economic Data page instead of three chart grids, one Calendar instead of three event lists. Scrapers/generators are still one-per-source internally (see Data flow above); the merging into unified pages happens once, in `core/generate_dashboard.py`, not in the underlying pipeline.

- **Cockpit** — Market Overview only (Equities/Interest Rates/Exchange Rates/Commodities cards). No tabs — this is the one page that stayed single-purpose, since "market data" doesn't fit the news/data/calendar shape the other three sources share. Commodities: Brent Crude, Iron Ore, Natural Gas, Diesel, Gasoline, Gold, Silver, Copper, Aluminum, Nickel, Jet Fuel (11 total — the old Suzano/Klabin pulp-price proxies were removed, see §4). Every card on every group — not just Commodities — carries a small "ⓘ" icon (hover tooltip, `ICON_INFO`) showing exactly which data provider it came from (`item.source` in `core/generate_dashboard.py`'s `_ck_make()`).
- **News** — unified feed: Portfolio + Brazil + China + Credit + GDELT articles, one list sorted strictly by recency (never resorted by importance). Filters: search box, **Country** (Brazil/China — Portfolio and Credit are both tagged Brazil since portfolio companies are B3-listed and credit news is the Brazilian credit market), **Sector** (one flat merged list across all sources — Credit's categories like "Recuperação Judicial" sit in the same dropdown as Brazil's/China's sector names), **Company** (populated only from the ~10 portfolio-tracked companies — the only source with company-level data), **Source** (multi-select checkbox list, e.g. select CNN Brasil + Valor Econômico simultaneously — `news_sources` is every distinct `a.source` value across the whole unified feed, ~174 with China's feed included since that one isn't domain-whitelisted; the checkbox list has its own client-side search-within-filter to stay usable at that count, separate from the main headline search box). **News Clipping**: "Generate Clipping" button in the feed header → checkboxes appear on every card → "Export Excel" downloads a real `.xlsx` (Date/Title/Link, Link as a genuine clickable hyperlink) of just the selected articles — see §4 for the implementation.
- **Economic Data** — unified chart grid: all Brazil (BCB) + China (chinadata.live) + Credit (BCB) dataset cards in one flat grid, client-side-tagged by source and filterable via pills (All/Brazil/China/Credit). The merge happens in the browser (`buildEconCharts()` in the template's JS combines the three `*_DATASETS` JS globals with a source tag) — no Python-side data model change was needed for this page.
- **Calendar** — unified event list: Brazil + China events, with Portfolio's (Cockpit's) earnings/COPOM events folded into the Brazil bucket. Filter: Country pills (All/Brazil/China).
- **Other Datasets** — the China dataset catalog browser (`buildCatalog()`), relocated to its own top-level page. Contents unchanged, stays China-only.

### Shared chart UI (standardized across every chart on every page)
- Every chart renders with Plotly's native modebar **off** (`displayModeBar:false`). Each card instead has small icon buttons in its header: **PNG**, **CSV**, **Expand**.
- **Expand** opens one shared modal (`#chartModal` / `.chart-modal*` CSS, `chartExpand()` in the template's JS) used by every chart — not a per-page component.
- Cockpit's FX cards (Exchange Rates group) are stat-only by design — big number + CSV download, no chart, no expand.
- The Economic Data page's chart grid is **one flat grid** for all 3 sources combined (no per-sector or per-source section breaks — see §4 for why); sector is shown via a colored left border + small tag per card, source via the pill filter, instead of hard row/page breaks.

## 3. Coding Standards

- **No comments explaining *what*** — code is expected to be self-descriptive via naming. Comments are reserved for non-obvious *why* (a workaround, a confirmed-bad external API, a subtle invariant).
- **Every scraper/event-generator file is self-contained**: each defines its own `HEADERS`, does its own `urllib3.disable_warnings()` + `sys.stdout.reconfigure()` at import time, computes `ROOT = Path(__file__).resolve().parent.parent`. This duplication is intentional — matches the project's "few files, each independently runnable" philosophy — except for genuinely cross-cutting concerns (HTTP retry logic in `core/net_utils.py`, importance-score bucketing in `core/scoring.py`, and the trusted-domain whitelist in `core/trusted_sources.py` — the last one specifically because 3 scrapers must apply the exact same rule, where duplication would risk drift).
- **All HTTP calls use `verify=False`** + `urllib3.disable_warnings(InsecureRequestWarning)` — required because the target corporate network (Itaú) intercepts HTTPS via a proxy with its own CA not in the `certifi` bundle. This is deliberate, not an oversight — do not "fix" it by re-adding cert verification without checking with Carlo first.
- **All HTTP calls go through `core.net_utils.get_with_retry()`** (2 attempts, backoff) rather than raw `requests.get()` — single-attempt calls were the root cause of a whole class of "0 articles fetched" bugs on flaky/corporate networks.
- **Windows compatibility:** every entry-point script calls `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` and every `subprocess.run()` passes `encoding="utf-8"` — Windows terminals default to CP1252 and choke on unicode (`→`, `—`, `…`) otherwise.
- **Timezone discipline:** all article/display timestamps must be in **Brasília time** (`America/Sao_Paulo`), not raw UTC from feed sources. Centralized in `parse_date()` in `core/generate_dashboard.py` — convert there, not at each call site.
- **No premature abstraction:** the 4 scrapers and 3 event generators are near-duplicates of each other by design (different queries/sectors) rather than parameterized into one generic module — this was a deliberate choice matching "few files, easy to reason about one at a time" over DRY-at-all-costs.
- **Testing:** none currently. No test suite, no CI. Verification during this project's development has been manual: run the actual scraper/generator, inspect real output counts, syntax-check generated JS with `node --check`.

## 4. Important Decisions

Chronological, most-recent last. Only decisions with lasting architectural consequence — see git log for the full incremental history.

- **Windows CP1252 fix** — `sys.stdout.reconfigure()` + `encoding="utf-8"` everywhere. *Why:* Windows terminal crashed on unicode output. *Impact:* every script touches this pattern at import time.
- **`verify=False` on all requests** — bypasses the corporate SSL proxy's untrusted CA. *Why:* certifi-based fixes didn't work; this is the bank's network doing TLS interception, confirmed by Carlo. *Impact:* `certifi` dependency removed entirely.
- **Fully static output, no server** — removed the refresh button and (later) deleted `core/server.py` entirely. *Why:* the real usage pattern is generate-then-email/upload, not "browse a running server." *Impact:* `dashboard.html` has zero `fetch()` calls; all chart/news data is baked in as inline JSON at generation time.
- **News window: 1 day → 7 days, fixed in two places** — raising `MAX_NEWS_AGE_DAYS` in `generate_dashboard.py` alone wasn't enough; each scraper had its *own* `MAX_AGE_DAYS=1` and a `balance_by_day()` that only bucketed "today" vs "yesterday", silently discarding anything older even after the render-time gate was widened. *Fix:* raised `MAX_AGE_DAYS` to 7 in all 4 scrapers + added a third "older" fallback bucket to `balance_by_day()` in 5 places (4 scrapers + `generate_dashboard.py`). *Lesson:* when a filter exists in multiple layers, check all of them.
- **Retry logic (`core/net_utils.get_with_retry`)** — single-attempt `requests.get()` calls were silently zeroing out entire sectors/datasets on transient timeouts. *Impact:* wired into every fetch site; bumped `update_dashboard.py`'s per-script subprocess timeout from 180s → 600s to give retries headroom without false-timeout aborts.
- **PIX BHKP/NBSK pulp proxies → Suzano/Klabin stocks** — FOEX's real pulp index is paywalled; the FRED series originally used (WPU0912, US PPI) was a meaningless proxy. *Fix:* replaced with `SUZB3.SA` / `KLBN11.SA` (B3-listed, liquid, fetchable via yfinance) as pragmatic proxies. **Superseded**: Carlo asked to remove these entirely (see the Cockpit commodities-expansion entry below) — no replacement pulp/paper metric exists on the Cockpit anymore.
- **Brazil 10Y: 12511 (broken) → Selic (wrong metric) → ANBIMA ETTJ (correct, real)** — first swapped the broken BCB series 12511 for the Selic policy rate as an honest-but-different fallback; Carlo rejected that outright ("the values of the brazilian interest is not actually the future curve but actually the selic, which is not the same thing"). Found ANBIMA's free daily ETTJ download (`https://www.anbima.com.br/informacoes/est-termo/CZ-down.asp?Dt_Ref=DD/MM/YYYY`) — a real, market-derived nominal yield curve, verified against independent reporting (fetched 14.44% vs. investing.com's reported 14.43% for the same day). The free endpoint serves one date per request and only retains a few months of history, so `fetch_br10y()` accumulates a local `data/br_ettj_history.json` (same merge/append pattern as the news caches): a short backfill on first run, then just the newest day appended each subsequent run.
- **File reorg: root-level `update_dashboard.py` + `setup.py`, `core/server.py` deleted** — matches actual usage: Carlo runs `update_dashboard.py` twice daily, `setup.py` once per machine, teammates only ever receive `dashboard.html`. Also deleted `templates/china_template.html` (610 lines, unreferenced dead file from before China was folded into the unified template).
- **Cockpit Market Overview: above news → below news → own "Market Data" tab (first, default-active)** — iterated per direct user feedback; settled on tab pattern to match Brazil/China/Credit's existing tab-first convention.
- **Chart UI standardization** — disabled Plotly's native modebar everywhere (was overlapping tiny 100px Cockpit cards); unified every chart card to the same PNG/CSV/Expand icon-button row; generalized the Cockpit-only "mkt-modal" into one shared `.chart-modal` used by all 4 pages. Also fixed a bug where the modal plotted its chart *before* being shown (`display:none` at plot time → zero-width render) — fixed by show-then-wait-two-frames-then-plot.
- **Removed per-sector section headers in Economic Data grids** — each sector previously started its own CSS auto-fill grid row, leaving visible blank slots whenever a sector's dataset count didn't evenly fill a row. Flattened to one grid per page; sector now shown via a colored left border + small tag instead of a hard row break.
- **Itaú logo embedded as base64** — resized the original 3840×3840 source (`Itaú.jpeg`) down to 120×120 (`assets/itau_logo.png`) before embedding, to avoid bloating `dashboard.html` by ~500KB for a header icon.
- **Timestamps converted to Brasília time at the source** — Google News RSS publishes in UTC; `parse_date()` (the single choke point used by all 4 news feeds) now converts via `zoneinfo.ZoneInfo("America/Sao_Paulo")` immediately after parsing, so every downstream display (article time, header "Updated" stamp) is correct without touching each call site individually.
- **News importance badges (Low/Medium/High)** — every scraper already computed a `relevance_score()` to decide what clears `MIN_SCORE`, then discarded it (`del a["_score"]`). Kept it instead: `core/scoring.py` buckets it (`importance_bucket()`, thresholds High ≥8 / Medium ≥5 / Low below, same on every page), each scraper persists it as `importance_score` + `importance`, and the template shows it as a small badge on every news card. Sort order stays strictly recency-based per explicit instruction — importance is informational only, not a resort key. Thresholds were calibrated against real re-scraped data, not guessed: every bucket has meaningful representation on all 4 pages (see git log commit `1e731d9`).
- **Brazil scraper: 7-day window → strict same-day, generic keywords → technical per-sector keywords** — Carlo wanted Brazil news restricted to today only (Brasília calendar date) with analyst-level search terms per sector instead of generic ones, while still hitting a 30-article floor per sector. Discovered via direct testing that Google News RSS ranks results by *relevance*, not recency, and caps each query at ~100 entries — a single broad OR-query per sector returned **zero** today-dated articles out of 100 results, because the query's relevance budget got spent on one dominant sub-topic. Fixed by querying each keyword *separately* (one request per term, not one OR-mega-query), which surfaced real same-day results the mega-query missed entirely. Confirmed empirically that some niche sectors (Pulp & Paper, Energy) genuinely don't have 30 same-day articles most of the time — the 30-floor is best-effort with score-based backfill (same-day candidates only, never reaching into other days), not a guarantee, and that's a real content-volume ceiling, not a bug.
- **Major reorg: 4 source-pages → 5 content-type pages** — Cockpit/Brazil/China/Credit (each reimplementing its own News/Events/Charts tabs) replaced with Cockpit/News/Economic Data/Calendar/Other Datasets, each internally filterable instead of hard-partitioned by source. Planned via `EnterPlanMode` given the scope (touches nearly all of `templates/dashboard_template.html` and `core/generate_dashboard.py`'s render pipeline); full plan and the resolved ambiguities (country tagging, no separate "topic" axis, Calendar scope) are in the approved plan — see git log commit for this reorg for the plan file reference. Key resolutions: Sector is one flat merged list (Credit's categories sit alongside Brazil's/China's, no separate "topic" filter); Portfolio and Credit news/events are tagged `country="Brazil"` (Portfolio companies are all B3-listed, Credit news is the Brazilian credit market); Portfolio's COPOM entries are deduplicated against Brazil's (both generators independently schedule COPOM with byte-identical titles) rather than shown twice. Scrapers/event-generators themselves are untouched — the merge happens once, in two new functions in `core/generate_dashboard.py` (`load_unified_news()`, `load_unified_events()`), keeping each source independently refreshable.
- **Source whitelist: trust as a scoring bonus → trust as a hard domain filter** — Carlo flagged that "many articles are from unreliable sources" and supplied a ~130-outlet whitelist (Brazilian financial/business/trade press, wires, government agencies, sector trade publications). Previously each of the 3 Brazil-market scrapers (`scraper.py`, `brazil_scraper.py`, `credit_scraper.py`) had its own `TRUSTED_SOURCES` set of outlet *names*, used only as a `relevance_score()` bonus (+3/+2) — an untrusted-source article could still clear `MIN_SCORE` and get kept. Replaced with a hard pre-filter: `core/trusted_sources.py` (new shared module, `TRUSTED_DOMAINS` + `is_trusted(href)`) is checked against `entry.source.href` — Google News RSS's real publisher-domain field, confirmed via direct testing (`entry.source == {"href": "https://valor.globo.com", "title": "Valor Econômico"}`) to be more reliable than matching on `entry.source.title` (matching `entry.link`, a `news.google.com` redirect URL, isn't possible directly). Any article whose domain isn't in the whitelist is discarded before scoring; the now-redundant trust bonus was removed from all 3 `relevance_score()` functions. **Kept as one shared module** (unlike each scraper's usual self-contained-duplication convention, see §3) specifically because all 3 must apply the identical rule — one list avoids drift if Carlo asks to add/remove a domain later. Deliberately **not** applied to `china_scraper.py` — the whitelist is 100% Brazilian outlets; applying it there would zero out the China feed entirely. The existing caches held articles fetched under the old (bonus-only) rule, and `merge_articles`'s freshness-based carryover doesn't retroactively re-check trust on cached rows — so the change wouldn't have taken full effect until those aged out (up to 7 days for Portfolio/Credit, same-day for Brazil). Purged all 3 caches (`data/news_cache.json`, `data/brazil_news_cache.json`, `data/credit_news_cache.json` — gitignored, regenerable, not source data) and re-ran fresh rather than waiting. Confirmed post-purge outlet lists are 100% whitelist domains. Verified via live run: `brazil_scraper.py`'s 30-per-sector floor is no longer met at all post-purge (Economy 5, Energy 2, Mining & Steel 2, Pulp & Paper 0, Agriculture 9, Financial 14, Industry 10, Trade & FX 49 — 91 total) — this is the same "real content-volume ceiling, not a bug" situation as the earlier same-day-only change, now compounded by the domain restriction; flagged to Carlo rather than silently loosening `MIN_SCORE` or the date gate to compensate, which would defeat the point of the whitelist. Portfolio dropped to 70 total across 10 companies (Bradesco: 0). Credit dropped to 29 total across 7 sectors.
- **Added GDELT as a second, independent portfolio-news source (`gdelt_scraper.py`)** — Carlo asked why Bradesco (one of Brazil's largest banks) returned 0 whitelist-passing articles and whether a better scraping method existed. Root-caused with live queries, not assumption (same standard as the Brazil same-day fix): `scraper.py`'s Bradesco query returns Google's full 100-result cap, but only 7 of those 100 fall inside the 7-day freshness window, and only 1 of those 7 was from a whitelisted domain — and that 1 scored below `MIN_SCORE` because its headline never said "Bradesco" (Google's phrase match is against the full article body, not the title). This is the *exact same* OR-mega-query relevance-dilution problem already diagnosed and fixed for `brazil_scraper.py` — `scraper.py` was never given the same per-keyword treatment, and a single-keyword test (`Bradesco BBDC4`) confirmed a full page of good, recent, trusted-domain results existed all along. The durable fix for that (per-keyword queries in `scraper.py`/`credit_scraper.py`) is still outstanding — GDELT was pursued as the *complementary* ask actually made. Live-tested GDELT's DOC 2.0 API (`api.gdeltproject.org/api/v2/doc/doc`) directly before writing any code: confirmed working query syntax (`sourcelang:portuguese`, `sourcecountry:brazil`, `sourcelang:english`), confirmed it returns the real publisher domain directly (`domain` field — no Google-redirect problem to work around), and confirmed its rate limit is real and unforgiving: a burst of unspaced test requests triggered a 429 lockout that was still active 30+ minutes later (confirmed via repeated, deliberately spaced re-checks — not a single data point), despite GDELT's own error message only stating "one request every 5 seconds." Built `gdelt_scraper.py` accordingly: throttles to 1 request/6s minimum, treats 429 as a hard stop (one retry after a 60s backoff, then abort and keep whatever was already fetched) rather than retrying aggressively into a worsening lockout. Scoped to **Portfolio companies only** (not Brazil/Credit sectors) for this first pass — same query taxonomy as `scraper.py` (reads `portfolio.xlsx` directly), two query legs per company (Portuguese restricted to `sourcecountry:brazil`, English unrestricted since international wires covering Brazilian companies aren't Brazil-sourced), each leg hard-filtered to a domain whitelist exactly like the Google News scrapers — GDELT itself does zero relevance ranking or quality filtering, so the whitelist matters *more* here, not less. Added `TRUSTED_DOMAINS_EN` to `core/trusted_sources.py` (Reuters, Bloomberg, FT, WSJ, CNBC, MarketWatch, Barron's, Forbes, Business Insider, S&P Global, Moody's, Fitch Ratings, AP, The Economist, BBC) since the existing `TRUSTED_DOMAINS` is Brazil-only and the English leg needed its own curated list — this restores the "international wires" tier that used to live locally in `scraper.py`/`credit_scraper.py` as a scoring bonus before the source-whitelist change above replaced it with a hard Brazil-only gate. Also added `DOMAIN_TO_OUTLET` (a reverse domain→outlet-name lookup, built from the same table `TRUSTED_DOMAINS` was normalized from) and `outlet_name()`, since GDELT's `domain` field is a bare hostname with no outlet title attached (unlike Google News RSS's `entry.source.title`). Output (`data/gdelt_news_cache.json`) is written in the exact same company-keyed shape as `news_cache.json` so it flows through the existing `flatten_news()` loader unchanged; merged into `load_unified_news()` as a 5th origin (`origin="gdelt"`, `country="Brazil"`), inheriting the same JS-assigned sector/company badge coloring as Portfolio rows (both lack a `color` field on the article dict, which is what the template actually branches on, not the origin tag). Wired into `update_dashboard.py`'s `STEPS` list right after the existing "Portfolio news" step. No new dependency — `requests` and `pandas` were already in `requirements.txt`.
- **`gdelt_scraper.py` redesigned for a ~150-company portfolio before it ever ran once at 10** — Carlo mentioned the final portfolio list will have 150 companies, not 10. Did the math before touching code: at 1 request/company/leg (the original design), 150 companies × 2 languages × the 6s throttle floor = **~30 minutes**, which blows `update_dashboard.py`'s 600s per-step timeout — and since the scraper only wrote its cache once at the very end of `main()`, a timeout kill would have discarded the *entire* run's progress, not just the excess. Asked Carlo two things before redesigning: whether the 150-company list would include tickers (**no — names only**) and how to handle the runtime problem (**"try batching companies per GDELT query first"**). Redesigned around both answers:
  - **Batched queries**: companies are grouped `BATCH_SIZE` (currently 5) at a time into one OR'd query (`("Petrobras" OR "Vale" OR ...) sourcelang:...`) instead of one request per company. This is **not empirically verified against GDELT's real query-complexity ceiling** — development testing tripped GDELT's rate limit before this could be confirmed live (see the rate-limit constraint below), so `BATCH_SIZE=5` is a conservative starting guess, not a measured safe value. Flagged prominently in the module docstring and as a `CLAUDE.md` follow-up (§7) — the first real run needs to be watched for silently thin/skewed per-company results, which would indicate the batch is too complex for GDELT to handle cleanly.
  - **No-ticker disambiguation**: added an optional `Ticker` column to `portfolio.xlsx` (blank is fine — falls back to the bare company name) instead of a hardcoded Python dict (`AMBIGUOUS_TERM`), which doesn't scale to 150 rows without silently degrading for any company someone forgets to add. Populated it for the current 10 companies as a working example. Article-to-company attribution (needed now that a batch's response is a shared pool across several companies) checks **both** the GDELT query term (which may be a ticker) **and** the plain company name as whole-word matches in the title — checking only the query term was tried first and caught by a quick offline unit test (a synthetic "Vale...Bradesco" ambiguous-batch case revealed articles using the company name rather than the ticker in the headline would go completely unattributed, silently undercutting GDELT's whole point). Zero or multiple companies matching in one article is dropped, not guessed at.

## 8. Known Constraints

- **BCB SGS series can silently 502** — confirmed with series 12511 (returned 502/timeout on every attempt while other series responded in <1s). Before wiring up a new BCB series, sanity-check it responds with a minimal `ultimos/1` query before trusting it in the pipeline.
- **ANBIMA's free ETTJ download only retains a few months of history** — `CZ-down.asp?Dt_Ref=DD/MM/YYYY` returns real data for roughly the last ~3-4 months and empty responses further back (confirmed by testing: mid-March 2026 onward worked, January/2025 dates didn't, from a "today" of July 2026). This is why `fetch_br10y()` accumulates its own history file rather than trying to bulk-backfill — there's nothing further back to backfill from, and it would be needlessly hammering ANBIMA's server to try.
- **FOEX pulp/paper index (PIX BHKP/NBSK) is paywalled** — no free public source exists. The Cockpit no longer shows a pulp/paper metric at all (the Suzano/Klabin stock-price proxy was removed at Carlo's request); if a real feed is ever licensed, a genuine pulp/paper commodity card could be added back.
- **Jet Fuel has no clean free outright futures price** — Yahoo Finance's `JET=F` is a differential/spread contract (oscillates near zero, sometimes negative), not an outright price; no other free source was found. The Cockpit's Jet Fuel card is an intentional, honestly-labeled N/A, not a bug — don't "fix" it by wiring up `JET=F` as if it were a real price.
- **Nickel has no Yahoo Finance ticker** — confirmed dead: `LNI=F`, `NI=F`, `NID=F`, `NICKEL` all 404/delisted. FRED's `PNICKUSDM` (monthly) is the only free source found; the Cockpit's Nickel card is intentionally lower-frequency than its neighbors, labeled `"FRED (monthly)"` rather than hidden.
- **Google News RSS is not an official API** — no SLA, no guaranteed rate limits, prone to transient timeouts especially over corporate networks. Retry logic mitigates but does not eliminate this.
- **ANP's fuel-distribution data runs ~2 months behind real time** — confirmed live (fetched mid-July 2026, latest complete month in the data was May 2026). This is ANP's own reporting cadence ("updated monthly until the last day of the month following the reference month"), not a bug in `fuel_scraper.py` — the Energy sub-page will always look slightly less current than the Cockpit/News/Economic Data pages, which are near-real-time. Don't "fix" this by trying to backfill more recent months that don't exist yet.
- **GDELT's rate limit is real and its lockout outlasts its own stated window** — the DOC 2.0 API's 429 response says "please limit requests to one every 5 seconds," but a short burst of unspaced test requests (during development, diagnosing this exact limit) triggered a lockout still active 30+ minutes later, confirmed via multiple spaced-out re-checks, not a single data point. Treat any 429 from GDELT as a signal to back off for a long time and stop, not to retry on a short timer — `gdelt_scraper.py`'s `gdelt_request()` already does this (one retry after 60s, then a hard abort for the rest of the run). Don't loosen this without a specific reason; a shorter/more aggressive retry policy risks a longer lockout on the next run too.
- **`dashboard.html` must remain a single self-contained file** — no external asset references, no `fetch()` calls. Any new feature that would require a live network call from the browser breaks the core distribution model (email/SharePoint a file, open with no server).
- **Corporate network intercepts HTTPS** — `verify=False` is required for the pipeline to run at all from the Itaú work machine. This is a known, accepted risk in this specific context (internal tool, non-sensitive public data sources only) — do not treat it as an oversight to silently "fix."
- **No test suite** — changes should be manually verified by actually running the affected script(s) and inspecting real output before considering a fix complete.

## 9. Project Rules

- **Never reintroduce a live server or a "Refresh" button in the UI** — the distribution model is generate-once, share-the-file. This was deliberately removed; don't add it back without explicit direction.
- **Never commit `dashboard.html` or `data/*.json`** — both are gitignored generated output. Only `data/portfolio.xlsx` (source data) is tracked.
- **Never relabel a data source as something it isn't** — if a metric can't be reliably fetched (e.g. Brazil 10Y), replace it with a real, correctly-labeled alternative rather than silently substituting different data under the original label.
- **Keep every chart's interaction pattern identical across pages** — PNG/CSV/Expand icon buttons, shared modal, `displayModeBar:false`. Don't introduce a page-specific chart UI variant without updating this file and reconciling the others.
- **Any new HTTP call must use `core.net_utils.get_with_retry()`**, not raw `requests.get()` — **except** where the target API has its own documented rate-limit contract requiring bespoke handling (see `gdelt_scraper.py`'s `gdelt_request()`: GDELT returns HTTP 429 with an explicit "one request every 5 seconds" message, which needs a long, deliberate backoff and a hard stop after one retry, not `get_with_retry`'s generic Timeout/ConnectionError-only retry loop, which doesn't inspect status codes at all and would either miss the 429 entirely or retry it too aggressively).
- **Any new age/freshness filter on news articles must be checked in every layer it exists** — this codebase has been bitten once by a filter existing in both the scraper and the renderer with different values.
- **Don't add `certifi`-based cert verification back** without confirming with Carlo — it breaks on the corporate network.
- **Don't reintroduce per-source pages** (a standalone "Brazil page," "China page," etc.) — the dashboard is organized by content type (News / Economic Data / Calendar / Cockpit / Other Datasets), each filterable by country/sector/company internally. This was a deliberate, explicitly-requested reorg away from per-source pages; don't revert to the old pattern without Carlo asking for it again.
- **Never test against GDELT with rapid/unspaced requests** — its rate-limit lockout has been confirmed to outlast its own stated "5 seconds" window by 30+ minutes. When diagnosing or extending `gdelt_scraper.py`, always throttle manually (6s+ between calls) even in one-off diagnostic scripts, not just in the production code path — a "quick test" burst can lock out the *next* real run too.
- **GDELT-sourced articles must pass the same domain-whitelist discipline as Google-News-sourced ones** — GDELT does no relevance ranking or source-quality filtering of its own (unlike Google News, which at least ranks by relevance), so skipping the whitelist for a GDELT-based feature would let through meaningfully lower-quality sources than the rest of the dashboard.
- **`scraper.py`/`brazil_scraper.py`/`credit_scraper.py` only keep articles from `core.trusted_sources.TRUSTED_DOMAINS`** — this is a hard filter checked against `entry.source.href`, not a scoring bonus; an off-whitelist article must be discarded, never merely down-scored. If Carlo asks to add/remove an outlet, edit `TRUSTED_DOMAINS` in `core/trusted_sources.py` once — don't touch the 3 scrapers individually, and don't apply this whitelist to `china_scraper.py` (it's Brazil-only and would zero out China's feed).

- **A government/agency statistical dataset's item hierarchy is not self-evident from item names — verify it against the actual numbers before treating sibling-looking categories as siblings.** FAOSTAT's "Other paper and paperboard" (item 1675) sounds like a residual leftover bucket; it's actually "Total minus Graphic papers," a supercategory that already contains Packaging and Household within it. Treating it as a fourth flat peer category alongside Packaging/Graphic/Household produced a structural-mix chart summing to well over 100%, with "other" as the *largest* slice — an obvious tell once looked at, but only because the actual percentages were checked, not just whether the code ran. Same standard as the `JET=F` negative-price catch: a plausible-looking number from a real, working data source can still be conceptually wrong, and a residual category being the dominant one (or a percentage breakdown not summing to ~100%) is exactly the kind of "does this pass the smell test" check worth running before trusting any newly-integrated classification hierarchy — one level up from checking that a *value* is sane, this is checking that a *category structure* is sane.
- **For a browser-generated downloadable file (News Clipping's `.xlsx`), `node --check`-clean JS syntax is not sufficient verification** — it proves the code parses, not that the *output file* is correct. The higher-value check: actually run the export logic against real/synthetic data with the real library (downloaded the pinned SheetJS bundle rather than trusting the CDN reference blindly), write the file, then *read it back* and assert on its contents — this is what caught nothing in this case (the logic was right first try), but it's the check that would have caught a wrong cell reference, a bad date format, or a hyperlink silently not attaching. Worth the extra few minutes whenever a feature's entire value proposition is "produces a correct file," since a syntax-clean function that writes a subtly-wrong file is a worse failure mode than one that visibly crashes.
- **The "category hierarchy isn't self-evident from names" lesson (FAOSTAT's "Other paper") recurred immediately on the very next external data source (BGS's "Iron and steel" group actually bundling three incompatible physical series)** — this isn't a one-off FAOSTAT quirk, it's a general property of official statistical datasets: broad, human-readable group labels are curated for reporting/publication, not for "safe to sum programmatically." Treat any new dataset's top-level grouping as a hypothesis to verify (pull a sample, check units, check whether sub-items' totals plausibly relate to the group total) before aggregating at that level, every time, not just the first time it bit.
- **Ticker mislabeling is a recurring, not one-off, risk when adding market-data symbols by "obvious" name** — `JET=F` (looked like jet fuel, was a spread contract), `GOLD` (looked like Barrick, was an unrelated company after Barrick's 2025 rename to "B"). Both were caught by checking `yf.Ticker().info`'s actual returned name/metadata against what was intended, not by the fetch succeeding. Company renames and ticker reassignments are exactly the kind of thing that makes a plausible-looking symbol wrong in a way that compiles/runs fine — always confirm the live `shortName` matches the intended company, especially for any ticker recalled from memory/training data rather than looked up fresh.
- **When the user explicitly asks for a concise plan and says "don't repeat what's in CLAUDE.md," honor that literally in the plan's structure, not just its prose density** — the Mining plan skipped re-explaining the by-then-established scraper/render/template pattern entirely and only covered what actually differed (data source, category scope, what's reused vs. newly duplicated). Matching the *scope* of a plan to what's actually new is a distinct skill from writing tersely — a well-trimmed plan for a 3rd instance of an established pattern should visibly shrink relative to the 2nd instance's plan, not just use shorter sentences.
